# Go + HTMX local web GUI implementation plan

ResearchForge has re-scoped the native Fyne desktop idea into a Go + HTMX local research cockpit. The CLI remains the authoritative automation path; the cockpit visualizes and reviews project state, guides small local workflow actions, and navigates artifacts generated by CLI workflows.

## Selected stack

ResearchForge uses **Go + HTMX** for the primary local web GUI. "Go HTMLX" in planning notes is canonicalized as Go + HTMX. The final purpose is a **local research cockpit**, not a passive artifact viewer.

Why this stack:

- keeps `rforge ui` Go-served and local-first;
- avoids a separate Node build chain for the primary app;
- reuses existing Go services and dependency-free `internal/ui` view models;
- supports progressive enhancement for forms, tables, artifact browsing, and guided local workflow actions;
- still allows targeted JavaScript visualization libraries for citation graphs, PRISMA diagrams, forest/funnel plots, and other chart-heavy views.

Alternatives rejected for the primary UI path:

- SvelteKit: strong for rich client-side dashboards, but adds a Node build chain and splits ownership between a Go backend and a larger frontend app.
- Astro: strong for static reports, but too static for project create/open, search/library, screening, and review workflows.

## Purpose boundaries

The cockpit should:

- open/create local Research projects through shared Go services;
- inspect papers, library records, source passages, and provenance links;
- navigate CLI-generated reports, PRISMA diagrams, citation graphs, evidence tables, and meta-analysis outputs;
- expose small local workflow actions where they improve review speed.

The cockpit should not:

- become the authoritative automation path over the CLI;
- hide provenance or accept unsupported scientific claims;
- move core research workflow logic into templates, partials, or browser JavaScript;
- require cloud services for local project review.

## Remaining implementation inputs

Before expanding the first HTMX slice, decide:

- local server/API boundary exposed by `rforge ui`;
- artifact loading rules for project-local papers, PRISMA diagrams, citation graphs, evidence tables, meta-analysis forest/funnel plots, and reports;
- visual QA expectations and screenshot/golden strategy;
- accessibility expectations for keyboard navigation, tables, charts, and graph alternatives.

## TDD slices

1. Add the Go + HTMX web workspace/templates/static assets and a minimal `rforge ui` local server entry point.
2. Wire project create/open to `internal/project` services.
3. Expose artifact APIs/files for papers, citation graphs, PRISMA diagrams, analysis outputs, and reports generated by CLI commands.
4. Wire dashboard, search, library, OSS, analysis, report/artifact, and guided-action pages/partials to `internal/ui` view models.
5. Add loading/error/empty states for each page using existing background job abstractions.
6. Add local-server smoke tests and DOM-level checks.
7. Keep all core logic outside templates/partials; HTMX pages should adapt view models and project artifacts only.

## Implemented dashboard (2026-06-16)

`rforge ui` now serves a real local research cockpit (`internal/webui.NewRouter`), not a placeholder:

- **Server + multi-port** — `rforge [--project <path>] ui [--addr :8080]` binds an `http.Server` (address also from `RFORGE_UI_ADDR`); multiple instances run concurrently on different ports, one per research folder. `rforge --json ui` reports the resolved address/project/routes without binding. Static assets are embedded (`internal/webui/static`).
- **Forge home timeline** — `/forge` shows the active project, current Forge state, latest provenance timeline, blocked review gates, background jobs, and next safe actions with CLI-equivalent commands shown for every guided action/button.
- **Spine workbenches** — `/workbenches` indexes HTMX/no-JS workbenches for source planning, import/dedupe, legal acquisition, parser arbitration, retrieval tuning, screening, evidence extraction, meta-analysis, report traceability, research map, connector health, and reproducibility/export.
- **Parser conflict review** — `/parsing` exposes the CERMINE adapter fallback policy, ordered parser candidates, reviewer-required conflict review, and the `rforge parse quality` CLI equivalent; conflicting fields are never auto-accepted.
- **Research map cockpit** — `/map` live-renders concept maps, citation neighborhoods, retrieval clusters, evidence coverage, filter/neighborhood controls, provenance overlays, keyboard-accessible alternatives, and a `/map/snapshot.json` export from the project knowledge graph.
- **Lab notebook timeline** — `/notebook` renders all project provenance events as human vs automated workflow events with counts and a `/notebook/snapshot.json` export.
- **Information architecture** — `/architecture` documents dashboard routes, partial endpoints, view models, no-JS fallbacks, background-job status artifacts, and ownership boundaries between HTMX templates and core Go services.
- **Permissions/privacy model** — `/privacy` documents local-only paths, copyrighted PDFs, reviewer notes, credentials, embeddings, cache files, and shareable report fields with default permissions, export rules, review gates, and UI behavior.
- **Source planning cockpit** — `/sources` previews the Meta-analysis spine source plan from a research question, covering OpenAlex, Semantic Scholar, Crossref, arXiv, PubMed/Europe PMC, NASA ADS, DOAJ/CORE, Unpaywall, Zotero/JabRef, and local imports with dry-run estimates, auth/privacy warnings, reviewer-approval language, and CLI-equivalent commands before network calls/imports/downloads.
- **Connector health/control center** — `/connectors` reads `<project>/data/source-live-smoke-snapshots/latest.json`, compares it with the connector capability registry, and shows missing/failing/stale/API-drift alerts before source connector use.
- **Dedupe/cluster review** — `/dedupe` provides a revtools-inspired identity-cluster review screen with matching rules, conflicting source fields, Zotero collection/tag context, citation-key preservation, reversible merge/split decision history from `data/identity-decisions.jsonl`, PRISMA counts, and identity/duplicate audit provenance.
- **Screening cockpit** — `/screening` provides HTMX views for active-learning queues, uncertainty/exploration flags, unresolved uncertain decisions, reviewer progress metrics, stopping diagnostics, and links to `data/screening-audit-bundle.json`.
- **Read papers in the browser** — `/papers` lists parsed documents from `<project>/parsed/`; `/papers/{id}` shows parsed sections/passages alongside a natively rendered project-local PDF (`/papers/{id}/pdf`) when one exists under `<project>/documents/`, falling back to parsed text only (copyright-safe). Paper ids are validated against a safe-stem allow-list to prevent path traversal.
- **Analysis** — `/artifacts` surfaces meta-analysis heterogeneity (I², τ², Q) and forest/funnel plot availability from `<project>/analysis/<run>-result.json`, plus PRISMA flow counts.
- **Knowledge graph** — `/artifacts` renders a deterministic server-side SVG citation graph from `<project>/data/citation-graph.json`, with nodes linking to their `/papers/{id}` reading page. A vendored, dependency-free `citation-graph.js` (offline, embedded) progressively enhances this into an interactive graph (pan/zoom/click) fed by the `/artifacts/graph.json` data endpoint; JS-disabled clients keep the static SVG fallback.
- **In-browser project switcher** — `/projects/switch` repoints the active folder at runtime (complements per-instance `--addr`), triggering HTMX refresh of the library and artifacts views.
- **Offline htmx** — htmx is vendored (`internal/webui/static/htmx.min.js`) and served from `/assets/htmx.min.js`, not a CDN. (The previous CDN `<script>` carried an invalid SRI hash that blocked htmx from executing, breaking every HTMX flow; vendoring restores offline, local-first operation.)
- **Browser e2e** — opt-in `playwright-go` tests (`internal/webui/playwright_e2e_test.go` and `internal/webui/playwright_spine_e2e_test.go`, build tag `playwright_e2e`, env `RFORGE_RUN_PLAYWRIGHT=1`, `make web-gui-e2e`) drive headless Chromium through shell nav, papers reader + native PDF embed, interactive-graph click-through and keyboard zoom, HTMX project switching, no-JS static-SVG fallback, and every Meta-analysis spine cockpit/workbench route with both JavaScript-enabled and JavaScript-disabled contexts. Excluded from the default gate and skip cleanly without browsers, matching the project's other opt-in e2e tests.
- **Screenshot regression** — `internal/webui/screenshot_e2e_test.go` (same opt-in harness) captures the shell, papers list, paper detail, artifacts page, and all Meta-analysis spine cockpit/workbench routes in a JS-disabled, fixed-viewport context. Core pages compare to committed golden PNGs (`internal/webui/testdata/screenshots/`) using a deterministic, AA-tolerant pixel diff (`imagediff.go`, unit-tested in the default gate); expanded spine pages are screenshot-smoked unless goldens are regenerated with `make web-gui-screenshots-update`. Screenshots are environment-sensitive and `*.actual.png` failure artifacts are git-ignored.

Decisions resolved for this build: multi-port = per-instance `--addr` flag **and** in-browser switcher; papers = native PDF + parsed text; graph = server-rendered SVG fallback **plus** vendored offline interactive JS (pan/zoom/click).

## Current ready seams

- `internal/ui` has dependency-free view models for dashboard, search, library, OSS, citations, screening, evidence, analysis, and reports.
- CLI commands already register report, PRISMA, citation graph, forest/funnel plot, evidence, and analysis artifacts in project state.
- `rforge --json ui` reports the selected Go + HTMX stack and ready implementation state for automation.
- `make web-gui-smoke` runs the Go + HTMX handler smoke tests in `internal/webui`.
